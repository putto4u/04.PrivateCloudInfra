## Helm 설치 및 환경 구성

쿠버네티스 운영에서 **Helm**은 복잡한 애플리케이션 리소스를 하나로 묶어 관리하는 패키지 매니저로, 실무 아키텍처 구성 시 필수적인 도구입니다. 현재 시스템에 Helm 커맨드가 인식되지 않으므로, 바이너리 설치를 통해 환경을 구성합니다.

### 1. Helm 바이너리 설치 (추천 방식)

`snap` 설치보다는 공식 바이너리를 직접 내려받아 경로를 지정하는 방식이 버전 관리와 시스템 안정성 측면에서 유리합니다.

* **설치 스크립트 실행**
```bash
curl -fsSL -o get_helm.sh https://raw.githubusercontent.com/helm/helm/main/scripts/get-helm-3
chmod 700 get_helm.sh
./get_helm.sh

```


* **설치 확인**
```bash
helm version

```



---

### 2. Helm의 핵심 아키텍처 이해

Helm은 **Chart**(설정 묶음), **Release**(실행된 인스턴스), **Repository**(저장소)의 세 가지 개념으로 구성됩니다. 이전 버전(v2)과 달리 v3부터는 클러스터 내 별도의 서버 컴포넌트(Tiller) 없이 사용자의 `kubeconfig` 권한을 직접 사용하므로 보안성이 대폭 강화되었습니다.

---

### 3. 초기 저장소(Repository) 설정

설치 직후에는 차트를 가져올 저장소가 등록되어 있지 않습니다. 전 세계적으로 가장 많이 사용되는 공식 아키텍트용 저장소들을 먼저 등록합니다.

* **ArtifactHub 공식 저장소 추가**
```bash
# Bitnami 저장소 (가장 표준적인 차트 제공)
helm repo add bitnami https://charts.bitnami.com/bitnami

# 저장소 업데이트
helm repo update

helm repo list

```

### 💡 실전 전문 팁 (Expert Tips)

* **Auto-completion 설정**: 실무 생산성을 위해 반드시 쉘 자동 완성 기능을 활성화하십시오.
```bash
echo 'source <(helm completion bash)' >> ~/.bashrc
source ~/.bashrc

```


* **유료 서비스 주의**: Helm을 통해 `AWS Load Balancer Controller`나 `ExternalDNS` 등을 배포할 때, 이와 연결된 **AWS NLB/ALB 생성 시 시간당 과금**이 발생합니다. 차트 배포 전 `values.yaml`에서 리소스 생성 옵션을 반드시 확인하십시오.
* **실수 방지**: `helm install` 시 `--dry-run --debug` 옵션을 사용하여 실제 클러스터에 반영하기 전 생성될 매니페스트를 미리 검증하는 습관이 중요합니다.
